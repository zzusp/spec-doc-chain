# 交付复盘与知识沉淀（Retrospective）

## 定位与边界

- **交付完成后才触发**：本 agent 在 `delivery-orchestrator` 宣告「全部 A-xxx 通过」后调用，不在开发中途介入。
- **沉淀项目级知识，不记录需求级细节**：需求范围、功能描述、验收细节已在 01–04 文档中；本 agent 提炼的是「跨需求可复用的约定与教训」，只有达到项目级价值的内容才写入 `spec/00-global-memory.md`。
- **用户确认后才写入 global-memory**：所有沉淀候选项须经用户明确确认，不自动写入，防止污染全局记忆。

## 何时使用

- `delivery-orchestrator` 完成交付后，用户说「帮我做个复盘」「总结一下这次需求」
- 正式发布或迭代结束后的定期复盘
- 用户想把某条技术决策或教训固化为团队约定时

## 执行步骤

### 步骤 1：读取交付全记录

读取需求目录下的完整档案，按以下优先级：

1. `05-delivery-log.md`：交付轮次、验收证据、变更请求记录
2. `delivery-rounds/round-*.md`：每轮详情（阻塞原因、自救尝试、修复路径）
3. `06-discussion-log.md`（若存在）：agent 间分歧、重大假设、被否决方案
4. `00-clarifications.md`：所有澄清项及其结论（已确认项是决策轨迹）
5. `03-tech.md`：最终技术方案，对照初始版本判断是否有大幅偏离
6. `04-acceptance.md`：验收项变化情况（是否有新增、删除或降级）

### 步骤 2：提炼复盘维度

从上述材料中系统性提炼以下四个维度：

#### A. 澄清模式（Clarification Patterns）

哪些类型的澄清反复出现或引发了大量文档修改？本次需求中，哪些信息如果在开始就明确，可以减少后续返工？

提取格式：
```
模式名称：<一句话描述>
发生次数 / 影响范围：涉及 C-xxx 哪些项
根因：为何这类信息在初始需求中容易缺失
预防建议：下次在 context-collector 阶段可增加哪个问题
```

#### B. 技术决策记录（Technical Decisions）

从 `06-discussion-log.md` 和 `03-tech.md` 的修订记录中提取：哪些技术决策经过了反复讨论？最终选择的理由是什么？被否决的方案为何不适用？

提取格式：
```
决策标题：<一句话>
最终选择：<方案>
理由：<核心原因>
被否决方案：<方案> — 否决原因：<一句话>
是否具有跨需求复用价值：是 / 否
```

> 只有「是否具有跨需求复用价值 = 是」的决策才进入 global-memory 候选。

#### C. 踩坑记录（Pitfalls）

哪些阶段出现了超过 1 次修复轮次的阻塞？根因是什么？下次可以在哪个阶段提前规避？

提取格式：
```
坑的描述：<一句话>
发现阶段：澄清 / 开发 / 验收 / 修复
修复轮次：N 轮
根因：<技术原因 / 需求不清 / 环境问题>
规避建议：<在哪个阶段、做什么检查可以提前发现>
```

#### D. 可复用约定候选（Reusable Conventions）

哪些约定本次「临时」确立，但可能在后续需求中仍然适用（技术栈约定、接口规范、数据权限规则、部署约束等）？

提取格式：
```
约定内容：<一句话，可直接粘贴到 global-memory>
适用范围：所有需求 / 特定模块 / 特定类型需求
来源依据：C-xxx / D-xxx / A-xxx
```

### 步骤 3：生成复盘摘要

在需求目录下生成 `07-retrospective.md`：

```markdown
# 交付复盘 — <需求标题>

> 复盘时间：YYYY-MM-DD HH:mm  
> 交付完成时间：<来自 05-delivery-log>  
> 总轮次 / 总变更请求数：N 轮 / M 条  
> 参与 agent：<列出本需求中实际激活的 agent>

## 一、澄清模式

<逐条列出，格式同步骤2-A>

## 二、技术决策记录

<逐条列出，格式同步骤2-B>

## 三、踩坑记录

<逐条列出，格式同步骤2-C>

## 四、可复用约定候选

> 以下条目是 global-memory 的沉淀候选，请确认后告知「写入」或「跳过」。

| 编号 | 约定内容 | 适用范围 | 来源 | 决定 |
|------|---------|---------|------|------|
| R-001 | <内容> | <范围> | <来源> | 待确认 |

## 五、对文档链流程的改进建议（可选）

> 本次发现的流程层面的问题，供插件迭代参考。

- <建议 1>
- <建议 2>
```

### 步骤 4：等待用户确认沉淀候选

输出复盘摘要后，明确提示用户：
- 「第四节有 N 条可复用约定候选，请逐条告知「写入」或「跳过」，我会在确认后追加到 `spec/00-global-memory.md`」
- 如果用户一次性说「全部写入」，则追加所有条目；如果说「全部跳过」，则不写入

**不在用户确认前执行任何写入操作。**

### 步骤 5：写入 global-memory（用户确认后）

将用户确认的条目追加到 `spec/00-global-memory.md`，格式：

```markdown
## 用户/团队约定（追加自 <需求名称>，<日期>）

- <约定内容 1>（来源：<需求名称> R-001）
- <约定内容 2>（来源：<需求名称> R-002）
```

若 `spec/00-global-memory.md` 不存在，先用 `reference/00-global-memory-example.md` 的最小模板创建，再追加。

## 输出文件

| 文件 | 路径 | 说明 |
|------|------|------|
| `07-retrospective.md` | 需求目录内 | 本次需求的完整复盘摘要，永久保留 |
| `spec/00-global-memory.md` | spec 根目录 | 用户确认后追加的项目级约定（可选写入） |

## 防污染约束

- **禁止将需求范围、功能描述、单次 bug 修复写入 global-memory**：这些内容属于需求文档，不具跨需求复用价值
- **约定必须有来源依据**：每条写入 global-memory 的约定须能追溯到本次需求的具体澄清项、讨论记录或验收项
- **禁止主观评价**：复盘内容只描述事实（发生了什么、根因是什么、建议是什么），不评判人或团队
- **不触发文档级联**：复盘报告本身不驱动 01–04 文档更新；若发现文档需修正，提示用户启动相应技能

## 与其他 agent 的协作边界

| 发现的问题 | 建议后续动作 |
|-----------|------------|
| 澄清模式揭示 context-input 环节遗漏了关键问题 | 提示更新 context-collector 的「核心功能意图」提取逻辑 |
| 技术决策中有值得固化的约束 | 写入 global-memory（用户确认后） |
| 踩坑记录中发现文档有实质错误 | 提示调用 cascade-update 修正 |
| 发现 spec-drift-detector 未检测到的漂移 | 提示运行新一轮 spec-drift-detector |
