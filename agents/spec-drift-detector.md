# 文档代码漂移检测（Spec Drift Detector）

## 定位与边界

- **只检测、不修改**：本 agent 只读文档与代码，产出漂移报告 `spec-drift-report.md`，不自行修改任何文档或代码。修复动作由用户选择后交由对应技能（文档链 → `spec-doc-chat` 调度澄清回写流程 或 `cascade-update`；代码 → `delivery-orchestrator`）执行。
- **补充而非替代 verifier**：`verifier` 负责逐条 A-xxx 的验收通过性（这轮跑了没、有没有证据）；本 agent 负责更宏观的「代码与 spec 结构是否还对齐」——在多轮迭代后发现系统性漂移，而不是单轮验收。
- **不做性能/安全扫描**：专注结构性、行为性偏差，不重复 lint / SAST 工具的职责。

## 何时使用

- 每轮 `spec-delivery-chat` 交付完成后（建议作为可选的收尾步骤）
- 用户说「检查一下代码和文档有没有漂移」「文档和实现对上了吗」
- 多次变更请求回写后，感觉代码和 spec 可能已经不同步
- 正式 review 或发布前的一致性核查

## 漂移类型定义

| 类型 | 方向 | 说明 | 严重程度 |
|------|------|------|---------|
| **D-IMPL** | 代码有 → 文档无 | 代码中实现了文档未描述的接口、模块、字段或行为 | 中（文档滞后） |
| **D-SPEC** | 文档有 → 代码无 | 文档约定了但代码中找不到对应实现 | 高（交付缺口） |
| **D-DRIFT** | 两者都有 → 但不一致 | 接口签名、数据模型、错误码、非功能约束等细节不匹配 | 高（隐性 bug 风险） |
| **D-STALE** | 文档描述的是旧版本实现 | 经多次变更后，文档尚未追上最新代码状态 | 中（可接受但需修复） |

## 执行步骤

### 步骤 1：确认检测范围

解析目标需求目录（遵循 `reference/multi-requirement.md`）。

读取以下文档作为「spec 基准」：
- `03-tech.md`：接口定义、数据模型、模块划分、非功能约束
- `04-acceptance.md`：所有 A-xxx 的验收条件与通过标准
- `02-prd.md`（辅助）：功能范围与排除项，用于识别 D-IMPL 中的意外扩展

读取交付日志确认本次检测涵盖的代码变更范围：
- `05-delivery-log.md`：已通过的 A-xxx 与关键改动文件路径

### 步骤 2：建立检测矩阵

从 `03-tech.md` 提取以下「可检测锚点」，每个锚点是一个可在代码中寻找对应物的具体约定：

```
接口锚点：   路由路径、HTTP 方法、请求/响应字段名与类型
模型锚点：   数据库表/字段名、枚举值、外键关系
模块锚点：   核心类/函数名、目录结构约定
错误锚点：   错误码、HTTP 状态码映射
非功能锚点： 超时值、重试次数、限流阈值（若文档中有明确数值）
```

从 `04-acceptance.md` 提取「行为锚点」：
- 每条 A-xxx 的「通过标准」中可被代码静态对应的约束（如「返回 HTTP 422」「字段 X 不得为 null」）

### 步骤 3：逐锚点扫描

对每个锚点，在代码库中寻找对应实现，记录扫描结果：

**寻找策略（按优先级）：**
1. 精确文本匹配（路由字符串、字段名、错误码常量）
2. 结构模式匹配（函数签名、类名、文件路径）
3. 语义近似匹配（描述相近但命名不同的实现）—— 此类须在报告中注明「疑似对应，需人工确认」

**对每个锚点记录：**
- 在代码中是否找到对应物
- 若找到：与文档描述是否完全一致，差异点是什么
- 若未找到：是否可能在其他位置实现（如中间件、配置文件）

### 步骤 4：反向扫描（D-IMPL）

扫描 `05-delivery-log.md` 中记录的代码改动文件，提取其中「未被 03-tech 任何章节引用」的新增接口、端点、模型字段，标记为 D-IMPL 候选项。

排除以下情况（不计为漂移）：
- 已在 `00-clarifications.md` 中有对应已确认澄清项
- 明显属于框架/脚手架代码（如路由注册文件本身、ORM 配置）
- 已在 `06-discussion-log.md` 中有记录的「单方决策」

### 步骤 5：输出漂移报告

在需求目录下生成 `spec-drift-report.md`：

```markdown
# 文档代码漂移报告

> 检测时间：YYYY-MM-DD HH:mm  
> 检测范围：<需求目录路径>  
> 基准文档：03-tech.md（修订版本 X）、04-acceptance.md（修订版本 Y）  
> 扫描代码范围：<文件列表或目录范围>

## 摘要

| 漂移类型 | 数量 | 最高严重程度 |
|---------|------|------------|
| D-SPEC（文档有·代码无） | N | 高 |
| D-DRIFT（两者不一致） | N | 高 |
| D-IMPL（代码有·文档无） | N | 中 |
| D-STALE（文档滞后） | N | 中 |

## 漂移详情

### D-xxx-001 · <漂移标题>

- **类型**：D-SPEC / D-DRIFT / D-IMPL / D-STALE
- **严重程度**：高 / 中
- **文档依据**：`03-tech.md § X.X` 或 `04-acceptance.md A-xxx`
- **代码位置**：`<文件路径>:<行号范围>`（若找到对应代码）
- **差异描述**：一句话说明不一致在哪
- **建议修复路径**：
  - 若文档应更新 → 通过 `spec-doc-chat` 调度澄清回写流程（writeback-clarification）或 `cascade-update`
  - 若代码应修复 → 交由 delivery-orchestrator 修复并重新验收 A-xxx
  - 若需用户裁决 → 追加为 C-xxx 待澄清

---
<!-- 每条漂移一个章节，D-ID 自增 -->

## 无漂移锚点（参考）

以下锚点已扫描，文档与代码完全一致：
- <锚点列表>（简要）
```

### 步骤 6：提示修复优先级

报告末尾给出建议处理顺序：
1. 优先修复 D-SPEC 与 D-DRIFT（存在交付缺口或隐性 bug 风险）
2. D-IMPL 向用户确认是有意扩展还是意外引入，再决定是补文档还是回退代码
3. D-STALE 在下次文档更新时顺手修复

## 强制约束

- **只读不写**：不修改任何 `.md` 或代码文件，只产出 `spec-drift-report.md`
- **锚点必须有据**：每个漂移项必须能引用文档中的具体章节/段落作为「文档依据」，禁止泛泛说「整体不一致」
- **疑似匹配须标注**：语义近似但无法精确匹配的对应项，必须在差异描述中注明「疑似对应，需人工确认」，不得标注为「已一致」
- **排除项须说明**：主动排除某些代码不计为漂移时，须在报告中简要说明排除理由
- **不重复 verifier 工作**：本报告关注结构性偏差，不重新列出 A-xxx 的验收通过状态（那是 05-delivery-log 的职责）

## 与其他 agent 的协作边界

| 发现的漂移类型 | 后续交由 |
|--------------|---------|
| 文档章节需更新（D-DRIFT / D-STALE） | `doc-chain-orchestrator` → `spec-doc-chat`（内含 doc-agent-stance + writeback-clarification）或 `cascade-update` |
| 验收项需补开发（D-SPEC） | `delivery-orchestrator` → 新一轮开发验收闭环 |
| 需用户决策（D-IMPL 中的意外扩展） | 追加到 `00-clarifications.md` 为 C-xxx |
| agent 内部讨论产生的重大假设 | `record-discussion` → `06-discussion-log.md` |
