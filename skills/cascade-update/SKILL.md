# 前置文档更新后级联更新后置文档

## 何时使用

- 用户刚编辑了分析报告、PRD、技术设计或验收清单中的某一文档，并希望「把后面文档也更新一下」。
- 用户明确说「我改了 PRD，请同步技术设计和验收清单」。

## 前置→后置关系

- 分析报告 更新 → 需检查并更新：PRD、系统技术设计、验收清单（若受影响）。
- PRD 更新 → 需检查并更新：系统技术设计、验收清单。
- 系统技术设计 更新 → 需检查并更新：验收清单。
- 验收清单 更新 → 无强制后置；若与 PRD/技术设计不一致，可提示用户。

## 路径约定与目标需求

- 路径、文件名约定详见 [reference/path-convention.md](reference/path-convention.md)。
- 多需求下目标需求目录解析顺序详见 [reference/multi-requirement.md](reference/multi-requirement.md)。

## 执行步骤

1. 确定被更新的文档类型及其对应文件路径（同一需求目录下的其他文档按固定文件名定位）。
2. 读取更新后的文档内容，归纳变更点（如范围、优先级、接口、验收条件等）。
3. 按上述关系依次处理每个后置文档：
   - 读取当前后置文档内容。
   - 根据变更点修改后置文档中受影响章节，保持术语与结论一致。
   - **Whole-document review（必须）**：写回前须执行 whole-document review（见 [reference/glossary.md](reference/glossary.md)）。
   - 写回文件。
4. 每更新一个后置文档后，在该文档的 **修订记录** 表中追加一行；调用 `extract-clarification`：对每个更新后的后置文档提取新待澄清项写入 `00-clarifications.md`，**支撑「通过用户不断澄清以不断完善文档」的核心理念**。**兜底**：提取时与 00 中已有项**去重**（同归属文档且关联章节重叠且**问题语义相似**则不新增），同一段落不重复插入，避免单次级联产生大量重复待澄清项或引发后续死循环。
5. 回复用户：列出已级联更新的文件、澄清文档中新增的待澄清项（如有），**并建议下一步**（如「请填写澄清文档中第 X 条后，说一声同步回 PRD」），推动澄清循环。

## 强制约束（必须/禁止）

- **后置与前置一致**：更新后的 03/04 中，功能列表、需求表述必须与用户给出的前置变更一致（如用户将 02 改为「用户登录、导出 PDF」，03、04 中必须体现这两项或明确对应），禁止与前置文档矛盾或随意发挥。
- **澄清项有据可查**：写入 00 的新增澄清项必须填写「归属文档」「关联章节」，且能与 03 或 04 的某小节/某表格对应，禁止空泛无来源。
- **04 与 03 可追溯**：验收清单中的 A-xxx 必须与 03 中的功能/设计点可对应（如 A-001 对应用户登录、A-002 对应导出 PDF），便于实现阶段追溯。
- **修订记录必追加**：每份被级联更新的文档（03、04、00）必须在修订记录中追加至少一行与本次级联或澄清提取相关的描述。

## 注意

- 只更新与前置变更**直接相关**的部分，避免无谓重写整篇文档。
- 若某后置文档不存在，可提示用户是否要先生成该文档（可结合 generate-doc-chain）。
- **仅修改验收清单（04）时**：虽无强制后置文档，仍建议执行「澄清提取」——分析 04-acceptance 是否有新待澄清项，若有则写入 `00-clarifications.md`，与其它文档更新行为一致。
