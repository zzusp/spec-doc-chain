# 统一对话入口

## 核心理念

**通过用户不断的澄清，从而不断完善所有工程文档。** 本技能调度的所有操作（生成、级联、提取待澄清、澄清回写）都服务于这一闭环：文档中的待确认点进入澄清文档 → 用户补充/确认 → 回写至工程文档 → 文档得以迭代完善。回复时应主动提示待澄清项并建议用户填写后执行回写，推动循环持续进行。

## 何时使用

- 用户在与规范文档链相关的对话中提出需求，包括但不限于：生成文档、更新某文档、同步澄清、级联更新、查看当前文档状态或待澄清项。
- 作为默认入口：当用户意图涉及「分析报告、PRD、技术设计、验收清单、澄清」且未明确指定技能时，由此技能解析并调度。

## 支持的操作与对应技能

| 用户意图示例 | 操作类型 | 调用的技能/动作 |
|--------------|----------|------------------|
| 生成整套文档、从分析报告开始生成、按顺序生成 | 按序生成 | generate-doc-chain |
| 从 PRD 开始生成、从技术设计开始补全后续文档 | 按序生成（指定起始文档） | generate-doc-chain |
| 分析现有项目/代码库、根据代码生成分析报告和PRD和技术设计 | 分析已有项目 | analyze-existing-project |
| 我改了 PRD，请同步后面的文档、更新技术设计和验收清单 | 级联更新 | cascade-update |
| 把澄清结论写回 PRD、根据澄清更新文档 | 澄清回写 | writeback-clarification（含 agent 表态流程，见下） |
| 把待澄清项加到澄清文档、从 PRD 里提取需要澄清的点 | 提取待澄清 | extract-clarification |
| 现在有哪些文档、哪些还没澄清 | 查看状态 | 读取文档列表与澄清文档，汇总回复 |
| 切换到 xxx、当前需求设为 xxx、列出所有需求 | 多需求管理 | 见下方「目标需求解析」与 reference/multi-requirement.md |

## 澄清回写的完整调度顺序（强制）

执行澄清回写时，必须按以下顺序调度：

1. 调用 `doc-agent-stance`：四个 agent 依次表态，产出修改决策表
2. 检查决策表：若存在冲突 / 时序依赖 / 新澄清触发，调用 `doc-agent-discussion`；否则跳至第 3 步
3. 调用 `writeback-clarification`：按最终决策表执行回写（跳过⏭文档，暂缓⏸文档）

表态与讨论的结果直接传递给 `writeback-clarification`，避免重复触发。若 orchestrator 已在上游完成步骤 1-2 并传入决策表，`writeback-clarification` 可跳过其内部步骤 0。

## 执行步骤

1. **解析意图**：从用户消息中识别上述操作类型及涉及文档（分析报告/PRD/技术设计/验收清单/澄清）、路径或范围。
2. **确认缺失信息与目标需求**：路径**默认**为 **spec/<YYYY-MM-DD>/<需求名称>/**。**多需求时**：先按 [reference/multi-requirement.md](reference/multi-requirement.md) 解析目标——用户显式指定 > 当前编辑文件所在目录 > spec/.state.json active > 仅有一需求目录则用该目录；否则列出 spec/ 下需求目录请用户指定。
3. **调度执行**：按对应技能说明执行（可在一个回复中串联多步，如先 cascade-update 再 extract-clarification）。
4. **回复结构**：
   - 先简短说明本次执行的操作类型与范围。
   - 若本次涉及澄清回写，在执行写操作前先展示「各 agent 表态摘要」与「修改决策表」，让用户看到哪些文档会被修改、哪些被跳过；若用户无异议则继续。
   - 若触发了结构化讨论，在决策表之后简要展示「讨论摘要」：每个冲突议题一行，格式为「议题 → 结论（收敛 / 升级为 C-xxx）」。
   - 再列出具体变更（生成/更新的文件、澄清文档变更）。
   - 最后给出下一步建议（如「请在澄清文档中填写第 3 条结论后，再说一声同步回 PRD」）。

## 强制约束（必须/禁止）

- **生成链回复**：当调度按序生成时，回复中必须明确说明生成顺序 01→02→03→04→00，并列出本次生成/更新的文件路径或清单，以及下一步建议（如待澄清条数、建议回写或级联）。
- **list 回复**：路径必须符合 `spec/YYYY-MM-DD/名称` 格式，且与 spec 下实际子目录一致；必须提及当前激活需求（来自 spec/.state.json）。
- **每步对应技能**：执行的操作必须与上述「支持的操作与对应技能」表一致，避免跳过级联或澄清同步。

## 原则

- **澄清优先**：生成或更新文档后，若有待确认点必须通过 `extract-clarification` 写入澄清文档；用户填写澄清后应及时调用 `writeback-clarification` 并提示下一步，不跳过澄清循环。
- **防死循环**：单轮（一次用户发起的回写/同步请求）内「回写 + 级联 + 澄清提取」执行一轮即止；新待澄清项由用户填写后显式发起下一轮。澄清提取时与已有项去重（同归属且关联章节重叠且问题语义相似则不新增），同段落不重复插入；回写后**仍须执行**澄清提取，仅经去重后不新增行即可。
- 一次对话可完成多步操作，但每步都要有明确对应技能与规则，避免跳过级联或澄清同步。
- 所有操作均可通过此统一入口完成，无需用户记忆具体技能名；若用户明确说「用 generate-doc-chain」，则直接执行该技能。
