# 按序生成文档链

## 何时使用

- 用户要求「生成整套文档」「从分析报告开始生成」「按顺序生成 PRD 和技术设计」等。
- 用户指定从某一文档类型开始，按前置顺序生成到验收清单或澄清。

## 文档顺序（不可跳过前置）

1. **分析报告**（`01-analysis.md`）— 需求/现状分析、结论与建议。
2. **PRD**（`02-prd.md`）— 产品需求文档，须引用分析报告中的结论与范围。
3. **系统技术设计**（`03-tech.md`）— 技术方案/架构与实现，须与 PRD 一致。
4. **验收清单**（`04-acceptance.md`）— 可验收条目（A-xxx），须覆盖 PRD 与技术设计。
5. **澄清**（`00-clarifications.md`）— 初始可为空或仅包含从上述文档中提取的待澄清项。

## 文档生成路径（强制）

- 输出路径、文件名约定与可选 `docs/` 根路径详见 [reference/path-convention.md](reference/path-convention.md)。
- 多需求下目标需求目录解析顺序详见 [reference/multi-requirement.md](reference/multi-requirement.md)。

## 全局记忆（须遵守）

- 若存在 **spec/00-global-memory.md**（项目级），撰写任一份文档前须**先读取**并将其内容作为全局约束纳入（技术栈、部署、团队习惯等）。若该文件不存在，可先创建并写入最小模板（见插件 `reference/00-global-memory-example.md`），再继续。单次需求的范围/结论不得写入全局记忆，应写在需求文档或澄清文档中。

## 文档模板（强制）

- 每份文档须按插件 `templates/` 下对应模板生成，保持章节与结构一致：
  - 分析报告 → `templates/01-analysis.md`
  - PRD → `templates/02-prd.md`
  - 系统技术设计 → `templates/03-tech.md`
  - 验收清单 → `templates/04-acceptance.md`
  - 澄清 → `templates/00-clarifications.md`
- 生成时填充模板中的占位与示例内容，不可删除模板中的一级/二级标题及 `<!-- CLARIFICATIONS:START -->` 等区块；每份文档须包含 **## 修订记录** 表，每次更新时追加一行。

## 执行步骤

1. 确认需求名称与日期，得到目标目录 **spec/<YYYY-MM-DD>/<需求名称>/**；若目录不存在则创建。
2. 若存在 `spec/00-global-memory.md`，先读取并作为全局约束。若目标目录下存在 `context-input.md`（由 context-collector-agent 产出），优先读取并作为分析报告与 PRD 的需求输入依据；其中 CC-xxx 待澄清项在生成 `00-clarifications.md` 时转为 C-xxx 编号写入。
3. 确认用户指定的起始文档类型（默认从分析报告开始）。**若起始文档不是分析报告（01）**：须确认该需求目录下所有前置文档已存在（如从 03 开始则需 01、02 已存在）；否则提示用户先生成前置文档或从分析报告开始生成。
4. 从起始文档开始，按顺序生成每一份文档：
   - 读取对应模板；**生成前**读取该需求目录下所有已存在的前置文档及已确认澄清（`00-clarifications.md` 中状态为已确认的项，若 00 已存在），保证引用与术语一致。
   - 基于模板框架填写内容。
   - **Whole-document review（必须）**：写回前须执行 whole-document review（见 [reference/glossary.md](reference/glossary.md)）。完成后写回该文档并在修订记录表中追加一行。
   - **支撑核心理念**：若存在需用户确认或二义性内容，须写入澄清文档。若该需求目录下尚无 `00-clarifications.md`，可在本轮到「生成澄清文档」一步时按模板创建并写入此前各文档积累的待澄清项；若 00 已存在则直接插入对应行。
5. 全部完成后列出文档链清单，**并明确提示当前待澄清项及建议**（如「澄清文档中有 N 条待确认，填写后可说一声同步回 PRD」），推动澄清循环。

## 强制约束（必须/禁止）

- **02 须引用 01**：PRD 正文中必须明确引用分析报告（如「见 01-analysis 结论」「分析报告中…」或引用 01 的章节名），不得与 01 脱节。
- **关键节非空**：01 的「结论」、02 的「需求范围」或等价节、03 的「实现目标」或等价节必须填写实质内容，禁止仅保留「待补充」等占位符。
- **04 可测试性**：每条验收项必须同时包含「验收步骤」与「通过标准」；通过标准禁止使用「功能正常」「体验良好」等不可断言表述，须可观测、可判定。
- **00 归属文档枚举**：澄清文档表格中「归属文档」列取值仅可为 `analysis`、`prd`、`tech`、`acceptance`、`global` 之一，与模板一致。
- **回复须列清单与下一步**：全部完成后回复中必须列出本次生成/更新的文件路径或清单，并明确提示当前待澄清项数量及建议下一步（如填写澄清后回写）。

## 输出约定

- 每份文档使用模板中的标题与章节结构；验收清单验收项编号为 A-xxx，便于与澄清项关联。文档末尾保留修订记录表。
- **04-acceptance 可测试性**：每条验收项的验收步骤须可执行、通过标准须可断言（禁止仅用「功能正常」等不可验证表述）；可与 PRD/技术方案中的 R-xx 或功能点对应，便于实现阶段 TDD。验收项须按模板含测试类型与优先级、验收目标与范围、前置条件（含数据准备）、验收步骤、通过标准、自动化测试支持、失败处理。**删除约束**：删除/清理只允许针对本用例自己创建或需求中明确标记为可删除的数据；验证删除功能时，数据准备须说明先添加要删除的测试数据或用户明确给出的可删数据。
