# 文档 Agent 结构化讨论

本技能是第一步（表态）与最终回写之间的协商环节。目标是让 agent 直接沟通解决冲突，而不是把所有分歧都丢给 orchestrator 或用户。

---

## 何时触发

**必须触发**（存在以下任一情形时，doc-agent-stance 步骤5 移交到本技能，而不是直接进入回写）：

| 触发情形 | 说明 |
|---------|------|
| 决策表中存在冲突 | A 表态「不需要修改」，但 B 在「对其他文档的影响」中指出 A 的某章节会联动变更 |
| 存在执行时序依赖 | 某文档的修改方向依赖另一文档的修改结果（决策表「执行时序」列有「待 XX 完成」） |
| 有新澄清触发 | 某 agent 在修改方向中注明「建议追加澄清项 C-xxx」 |

**不触发**（以下情形直接进入回写）：

- 决策表无冲突、无时序依赖、无新澄清触发
- 所有 agent 表态一致且修改方向清晰

---

## 执行步骤

### 步骤 1：识别议题，分类处理

读取 doc-agent-stance 产出的决策表，提取所有需要讨论的条目，按类型分类：

| 类型 | 识别方式 | 处理路径 |
|------|---------|---------|
| **冲突项** | A 说不改，B 说 A 要联动改 | 进入讨论轮次（步骤2） |
| **时序依赖项** | 执行时序列有「待 XX 完成」 | 进入顺序确认（步骤3） |
| **新澄清触发项** | 修改方向含「建议追加 C-xxx」 | 直接写入澄清文档（步骤4），不进入讨论 |

**议题锁定**：每个讨论议题必须对应决策表中的一个具体条目，禁止在讨论中引入与当前澄清事实无关的新议题。

**处理顺序**：若同时存在多种类型，按以下顺序执行：新澄清触发项（步骤4，直接写入，不阻塞）→ 冲突项（步骤2）→ 时序依赖项（步骤3）。原因：冲突讨论结果可能改变文档的执行修改状态，进而影响时序排列；先解决冲突，再确认时序，避免时序需要二次调整。

---

### 步骤 2：冲突项讨论（收敛驱动 + 动态轮次上限）

讨论以**收敛状态**为驱动：每轮结束后 orchestrator 先判断是否已收敛，**收敛即立即停止**，不跑满上限轮次。未收敛时，按分歧类型决定是否继续下一轮或升级裁决。

#### 分歧类型与轮次上限

| 分歧类型 | 上限轮次 | 说明 |
|---------|---------|------|
| **表述差异**（内容一致，措辞不同） | 0 轮 | orchestrator 直接裁决措辞，不进入讨论 |
| **单文档内影响范围分歧** | 2 轮 | 单一文档的章节覆盖范围争议 |
| **跨文档联动冲突** | 3 轮 | 涉及多文档依赖链的分歧，需要更多轮次对齐 |
| **业务边界 / 需求范围分歧** | 2 轮后升级 | agent 无权自行裁决，尽早升给用户 |

**全局兜底**：任意单议题总轮次 ≤ 4，防止边界条件导致无限循环。

orchestrator 在步骤1识别冲突议题时，同时标注每个议题的分歧类型，并据此设定该议题的轮次上限。

#### 发言格式

```
[发起方 → 接收方] 第N轮
- 议题：引用决策表中的具体条目（如「01-analysis.md 执行修改列」）
- 我的立场：需要修改 / 不需要修改 / 修改方向调整为 XX
- 依据：引用澄清事实或文档中的具体内容，一句话，禁止泛化
- 期望对方确认：一个明确的问题或行动
```

#### 回应格式

```
[接收方 回应] 第N轮
- 接受 / 不接受 / 部分接受
- 理由：一句话，有据可查
- （若不接受）我的修正立场：具体说明
```

#### 每轮结束后：收敛检查（Early Stop）

每轮发言+回应完成后，orchestrator **立即执行收敛检查**，满足以下任一条件即视为**已收敛**，停止该议题的讨论并更新决策表，不再启动下一轮：

| 收敛条件 | 说明 |
|---------|------|
| 接收方明确说「接受」 | 接受后的决策与发起方立场一致 |
| 双方达成修正方案 | 与原决策表不同，但双方均接受 |
| 发起方撤回异议 | 「理由充分，我撤回联动提示」 |

未收敛时，检查当前已用轮次是否达到该议题的上限：
- **未达上限**：N++ 继续下一轮
- **已达上限**：进入步骤5（升级裁决或 orchestrator 裁决）

#### 轮次计入规则

- 仅计入含**实质性立场变化或新论据**的轮次，纯粹确认、重复表述不计入轮次
- 多冲突项联动检查：每个冲突项讨论结束并更新决策表后，orchestrator 检查剩余冲突项的判断条件是否因此发生变化；若有变化，更新对应条目，再处理下一冲突项

#### 禁止发散

- 发言内容只能聚焦当前议题，每条发言不超过 4 行
- 禁止在讨论中引入与当前澄清事实无关的新议题

---

### 步骤 3：时序依赖确认

对于执行时序依赖项，不需要讨论，只需 orchestrator 确认顺序并在决策表「执行时序」列中明确标注：

```
第1步（无依赖，可立即执行）：02-prd.md、03-tech.md
第2步（待第1步完成后执行）：04-acceptance.md
```

若某文档的修改内容依赖另一文档的新澄清结论（如 04 依赖 C-003 确认后才能写断言），则该文档标注「暂缓，待 C-003 确认」，不阻塞其他文档的执行。

---

### 步骤 4：新澄清项直接写入

对于修改方向中触发的新澄清（技术可行性存疑等），orchestrator 直接在 `00-clarifications.md` 中追加新行：

```markdown
| C-xxx | 待确认 | 高/中/低 | [影响范围] | [归属文档] | [关联章节] |
| 由 [agent名] 在讨论中提出：[具体问题] | | |
```

追加后告知用户，不阻塞当前其他文档的回写执行。

---

### 步骤 5：未收敛分歧的升级裁决

**触发条件**：某冲突项达到该议题轮次上限后仍未收敛。

按分歧类型决定处理方式：

| 分歧类型 | 到达上限后的处理 |
|---------|--------------|
| 跨文档联动冲突（技术细节） | orchestrator 以「有据一方」为准裁决，tech-agent 有最终决策权 |
| 业务边界 / 需求范围分歧 | 升级为待澄清项，写入双方立场摘要，交由用户裁决 |
| 需求优先级分歧（本期是否包含） | 升级为待澄清项 |
| 技术方案存在互斥选项 | 升级为待澄清项，注明各方案的文档影响 |

**不允许升级的情形**（必须在讨论中自行收敛）：

| 情形 | 处理要求 |
|------|---------|
| 仅表述差异，内容一致 | orchestrator 直接裁决措辞，0 轮讨论 |
| 技术实现细节分歧 | tech-agent 有最终决策权，orchestrator 直接裁决 |
| 某 agent 仅「不确定」但无实质论据 | 以「有据一方」为准裁决，不升级 |

**升级格式**：将分歧写入 `00-clarifications.md` 新行，**必须包含讨论摘要**：

```markdown
| C-xxx | 待确认 | 高 | [影响范围] | global | [关联章节] |
| 讨论未收敛，需用户裁决：[一句话描述分歧核心]。
  背景：[agent A] 认为 [立场A，一句话]；[agent B] 认为 [立场B，一句话]。
  两种选择的影响：选A则[影响]，选B则[影响]。 | | |
```

写入后：
- 将该冲突项在决策表中标注「⏸ 待用户裁决（见 C-xxx）」
- 暂停该文档的修改，不阻塞其他文档的执行
- 告知用户「有 1 处分歧需要您裁决，详见 C-xxx」

---

### 步骤 6：输出更新后的决策表

讨论结束后，输出最终决策表，标注每个议题的处理结果：

```
📋 讨论完成，更新后的决策表（C-xxx：[澄清摘要]）：

✅ 执行修改（3 份）：01-analysis.md、02-prd.md、03-tech.md
⏭ 不需要修改（1 份）：04-acceptance.md
⏸ 暂缓（0 份）：—
🆕 新增待澄清（1 条）：C-003（Kafka 方案可行性）

| 文档 | 最终结论 | 执行修改 | 影响章节 | 执行时序 | 讨论备注 |
|------|---------|---------|---------|---------|---------|
| 01-analysis.md | 需要修改 | ✅ | §矩阵 | 第1步 | 讨论后 analysis 接受联动 |
| 02-prd.md | 需要修改 | ✅ | §R-08 | 第1步 | 无争议 |
| 03-tech.md | 需要修改 | ✅ | §接口 | 第1步 | 追加 C-003 待确认 |
| 04-acceptance.md | 需要修改 | ✅ | A-004 | 第2步（待03完成） | 时序依赖03 |

👉 建议下一步：
- 立即执行第1步回写（02-prd、03-tech、01-analysis）
- 第1步完成后执行第2步（04-acceptance）
- C-003 已写入待澄清，请您确认后再执行 03-tech 的方案部分
```

移交 `writeback-clarification` 按时序执行回写。

---

## 强制约束

- **收敛优先**：满足收敛条件时立即结束该议题，不因「还没到上限轮次」而继续
- **讨论不替用户决策**：agent 讨论只解决文档层面的影响范围与修改方向，不裁决业务需求本身
- **有据才能发言**：每条发言必须引用决策表中的具体条目或文档中的具体章节，禁止空泛表达「我觉得」「可能会」
- **动态上限硬约束**：按分歧类型设定上限（表述差异0轮、单文档2轮、跨文档联动3轮），全局兜底任意议题 ≤ 4 轮
- **升级需有讨论摘要**：升级为待澄清时必须在澄清项中写明双方立场与各自影响，不允许直接抛给用户一个空问题
- **不遇难即抛**：仅当经过完整讨论且分歧属于业务决策范畴，才升级；技术细节、表述差异、单方无据异议一律在讨论中收敛

## 路径约定

- 本技能在 `doc-agent-stance` 之后、`writeback-clarification` 之前调用
- 角色定义见：`reference/doc-agent-roles.md`
- orchestrator 在本技能中担任**主持人**角色：控制发言顺序、判断收敛/未收敛、执行升级写入
- 目标需求目录解析：同 `reference/multi-requirement.md`
