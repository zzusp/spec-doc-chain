# 交付流程入口（实现与验收闭环）

## 定位与边界

- **不同于 `spec-doc-chat`**：本技能面向“实现阶段”，允许并鼓励**修改代码、增加测试、运行验收**；`spec-doc-chat` 及其调度链条只负责撰写/同步文档，不进行代码改动。
- **唯一完成标准**：`spec/**/04-acceptance.md` 中所有验收项（A-xxx）均在本流程内由你**执行 + 观测 + 判定 + 证据落盘**并给出“通过”；在此之前禁止宣称交付完成。
- **验收责任不可转嫁（原则）**：交付结束时，不得存在任何“仍需用户执行某操作才能完成的验收或验证”。即：每个 A-xxx 的验证闭环必须由你在本流程内完成（你发起执行、读取结果、给出通过/不通过、写入日志）；若某验收项需用户“跑脚本、设环境、点一下、确认一下”等才能算验证完成，则该验收项视为**未在本流程内通过**，交付不得结束。不限定具体措辞——凡语义上把验收/验证责任留给用户或推到“用户侧再执行”的，均属禁止（如：建议用户验证、请用户执行、待用户重跑、需用户先设置 XX 再验证、已知限制为“依赖用户鉴权”却仍写交付结论等）。
- **用户满意优先**：即使所有 A-xxx 都通过，若用户反馈“与预期有偏差/需要调整”，应将其视为**变更请求**进入新一轮闭环（见下文「变更请求」）；直到用户确认满意并验收通过，才算最终交付。

## 何时使用

- 用户已通过 `spec-doc-chat` 完成（或基本完成）该需求的文档链，尤其是已具备可执行的 `04-acceptance.md`。
- 用户明确要求“按文档开始开发/交付/跑验收/修复直到通过”等。

## 输入与前置检查（必须）

1. **解析目标需求目录**（写操作前置）：按 `reference/multi-requirement.md` 的优先级解析目标目录（用户指定 > 当前编辑文件所在目录 > `spec/.active` > 仅有一需求目录 > 否则列出并要求用户明确）。
2. **读取交付依据**（同一需求目录下）：
   - `02-prd.md`（范围与约束）
   - `03-tech.md`（实现方案与关键约束）
   - `04-acceptance.md`（A-xxx、验收步骤、通过标准）
   - 若存在：`00-clarifications.md`（仅将“已确认”的结论作为约束）
   - 若存在：`05-delivery-log.md`（在其末尾继续追加，不覆盖）
3. **可执行验收预检（必须）**：读取 `04-acceptance.md` 后，逐条 A-xxx 检查：是否有可执行步骤、可观测输出、可断言通过标准；是否隐含「需用户再执行/再确认」才能完成的验证。不满足的项在交付日志中记录为「待补齐可执行步骤/证据采集方式」，并给出具体修改建议（不降低通过标准）；再在不修改验收标准的前提下尽量补齐可执行步骤与证据方式（必要时提示用户补充环境/数据；但不要用“无法验证也算通过”替代）。

4. **环境/启动/编译阻塞的处理（强制自救门槛）**：
   - 任何“无法编译/无法启动/无法重启/依赖不满足”都属于交付闭环的一部分，必须先由本流程尝试解决，而不是立刻抛给用户。
   - 在请求用户介入（例如让用户去编译、去重启、去装 JDK）之前，必须在本流程内完成**至少 3 次不同方向**的自救尝试，并把每次尝试的命令、错误与结论写入本轮的 round 文件（`templates/05-delivery-round.md` 中的自救表）并在 `05-delivery-log.md` 中摘要记录。
   - **自救方向限定为三类**（需尽量覆盖不同方向）：
     1. 依赖/编译层修复（如 npm/pip 安装、JDK/Node 版本调整、依赖冲突解决）——目标是让代码能够正常编译/启动。
     2. 配置/环境变量层修复（如 `.env`、端口、数据库连接串、鉴权配置）——目标是连上真实依赖环境。
     3. 代码层绕过/mock（仅限外部依赖）——目标是暂时隔离**外部系统不可达**导致的阻塞，只允许 mock 第三方 API、下游服务等外部依赖，**禁止 mock 本需求自身的核心业务实现逻辑**。
        - **进入本方向前必须**：读取 `04-acceptance.md` 的 `Mock 限制` 列，确认拟 mock 的对象未被标注为「禁止 mock」；若被标注禁止，则本方向不可用，直接上报阻塞等待用户介入，不得继续尝试 mock。
   - 只有当你确认阻塞属于用户侧不可替代条件（权限/网络/合规/缺少用户环境且本流程无法安装配置等），才允许中断并明确说明需要用户提供什么；随后在**同一交付流程内**继续验收闭环。

5. **Mock 污染标记规则（防止“表面通过”）**：
   - 任何验收项 A-xxx 只要在**依赖了方向③ mock 的环境**下执行了验收步骤，其状态在 `05-delivery-log.md` 的「验收项状态总表」中**必须标记为 `MOCK-BLOCKED`，禁止标记为“通过”**。
   - round 文件中必须记录：
     - 使用了哪个 mock（具体到类/接口/服务或外部系统标识）；
     - 该 mock 覆盖了哪些 A-xxx 的通过标准（若 mock 直接返回“通过”所需的响应/结果，则该轮的「通过」结论视为**无效**，只能记作 `MOCK-BLOCKED`）。
   - **存在任何 `MOCK-BLOCKED` 状态的 A-xxx 时，禁止给出最终交付结论**；交付只能处于「阻塞待真实环境验证」状态。
   - 当外部依赖恢复或用户提供了必要条件后，必须：
     1. 移除或停用对应 mock，用真实依赖重新执行所有 `MOCK-BLOCKED` 的 A-xxx；
     2. 重新收集证据（Expected vs Observed + DB/副作用核对）并写入新的 round 文件；
     3. 将这些 A-xxx 的状态从 `MOCK-BLOCKED` 改写为「通过」或「不通过」；
     4. 在 round 文件中显式记录为「Mock 解除回归」。
   - 只有当不存在任何 `MOCK-BLOCKED` 项且所有 A-xxx 均在真实依赖环境下通过后，才允许进入最终交付。

## 变更请求（交付中出现偏差/调整时的强制回流）

在实际交付过程中，用户可能提出“当前实现与预期不一致/需要调整”。此时你必须按以下顺序处理，并记录到交付日志：

1. **记录变更请求**：将用户原话、期望结果、影响范围、优先级写入 `05-delivery-log.md` 的「变更请求记录」。
2. **先对齐文档再改代码**（除非是明显 bug 修复且不改变范围）：更新同一需求目录下的文档，使其与用户最新预期一致：
   - `02-prd.md`：范围、交互/行为、约束、非功能需求
   - `03-tech.md`：对应实现方案的调整（存量结构如何改、兼容性/迁移策略等）
   - `04-acceptance.md`：把变更映射为可执行的验收项（新增/修改 A-xxx），并保证步骤与通过标准可断言
   - 若有待用户确认点，写入 `00-clarifications.md` 并按澄清流程闭合后再继续
3. **进入新一轮交付闭环**：以更新后的 `04-acceptance.md` 为准继续“开发→验收→不通过→修复→再验收…→通过”，直至用户确认满意并验收通过。

## 交付日志（强制落盘）

- **日志文件路径**：在目标需求目录写入或追加 `05-delivery-log.md`。
- **日志模板**：以 `templates/05-delivery-log.md` 为框架；若已存在则仅追加新轮次与更新表格，不重写历史。
- **轮次详情可拆分（推荐）**：为控制上下文长度与提升审阅体验，建议将每轮的详细“Assertions + 可见证据包 + DB Before/After + 环境自救尝试”写入独立文件（模板 `templates/05-delivery-round.md`），主日志 `05-delivery-log.md` 只保留：
  - 首页仪表盘
  - 验收项状态总表（固定表）
  - 轮次索引（总览 + 详情入口，含指向 round 文件的引用路径）
- **最小记录要求**（每一轮必须写）：
  - 本轮覆盖的验收项列表（A-xxx）
  - 本轮开发/修复的关键改动（文件路径级别）
  - 本轮真实执行的测试/验收命令与关键输出（证据）
  - 本轮结论：通过 / 不通过（不允许中间态）
  - 不通过时：根因与修复方案（对齐 PRD/技术设计）
  - **索引 + 详情必须同步**：更新主日志的「轮次索引」表时，必须同时产出/更新对应的 round 详情文件（例如 `delivery-rounds/round-<NN>.md`，使用 `templates/05-delivery-round.md`）；禁止只改索引表就结束。
  - **验收项状态总表必须更新**：每轮结束必须更新 `05-delivery-log.md` 中的「验收项状态总表」：对本轮覆盖的每个 A-xxx 写明当前状态（未开始/不通过/通过/阻塞）、最近验证轮次与可断言证据引用。
  - **每轮写入顺序（强制）**：
    1. 先更新主日志 `05-delivery-log.md`：首页仪表盘（当前进度/卡点/下一步）、验收项状态总表（本轮覆盖的 A-xxx）、轮次索引（新增 Round 行并填入详情文件路径）
    2. 再写本轮 round 文件 `delivery-rounds/round-<NN>.md`：本轮摘要 → Assertions → 可见证据包 →（如阻塞）自救尝试
    3. 最后回到主日志校验：状态总表与轮次索引没有遗漏/不一致

## 强制约束（必须/禁止）

- **验收项状态总表必完整**：`05-delivery-log.md` 中的「验收项状态总表」必须包含本需求 04 中所有 A-xxx，每项状态为「未开始/不通过/通过/阻塞/MOCK-BLOCKED」之一，并注明最近验证轮次或证据引用，禁止留空或遗漏。
- **证据必须落盘**：每轮 05 或对应 round 文件中必须包含本轮真实执行的命令或可观测输出摘要（如 `Test-Path README.md` 结果、接口响应关键字段），禁止仅写「已验收通过」而无可见证据。
- **禁止转嫁验收责任**：05 全文及对用户的交付结论中禁止出现「需用户执行」「请用户验证」「待用户确认后再验收」等将验收责任转嫁的表述；凡需用户再执行某操作才能算验收完成的，该 A-xxx 不得标记为通过。
- **轮次索引与详情一致**：若存在 round 文件，主日志「轮次索引」中必须有一条指向该 round 文件（路径或文件名一致），禁止只更新索引表而不产出或更新对应 round 详情文件。

### 上下文长度控制（必须遵守）

- **读日志时**：主日志只读取「首页仪表盘 + 验收项状态总表 + 变更请求记录 + 轮次索引」，需要细节时再按需读取最近 1～3 个 round 文件；更早轮次仅参考索引与状态，不把历史详情整段带入上下文。
- **写日志时**：坚持“摘要 + 可复现证据”：
  - 只贴关键输出片段（建议 10～30 行以内），其余用截断说明，避免把长响应体/全量控制台输出写入日志
  - 不在日志里重复粘贴整份验收清单或大段文档正文，改为引用（A-xxx、小节标题、文件路径）
  - 每轮优先更新主日志的「验收项状态总表」与「轮次索引」，再写 round 详情文件；若采用 `delivery-rounds/` 拆分轮次文件，则可适度放宽单轮长度以保证证据完整，但必须保持结构化、分段与可读
  - **可读性优先**：每轮必须先写「本轮摘要」（1 分钟可读完），证据集中写在“证据清单（Assertions）”，命令可复制、断言用 Expected/Observed 对照表表达，避免散乱长段落

## 强制流程（不可跳过）

你必须严格执行并在每轮内记录日志：

开发实现（建议 TDD：RED→GREEN→REFACTOR）
→ 测试 / 验收（必须真实执行）
→ 结论（默认不通过）
→ 若不通过：定位根因 → 修复 → 回到“测试 / 验收”
→ 重复直到所有 A-xxx 在本流程内执行并通过（无任何项留给用户执行或确认）
→ **最终交付**（仅当不存在任何“需用户执行/确认”的验收项时方可填写；交付结论与回复中不得遗留由用户完成的验证步骤）

## 执行策略（推荐）

- **以验收项为粒度推进**：优先 P0，再 P1、P2；一轮可覆盖 1~N 个验收项，但每个验收项必须有明确结论与证据。
- **证据优先**：任何“无法运行/无法观测/缺少数据”的情况都视为不通过，并记录原因与下一步所需条件。
- **反表象验收（强制）**：禁止把 “HTTP 200 / code=200” 当成通过标准；每个 A-xxx 必须记录“期望 vs 观测”，并至少给出三类证据：接口/返回值 + 业务结果（必选） + DB/副作用（至少其一）。缺少业务结果或 DB/副作用核对（若涉及写入/更新/删除）一律不通过。
- **证据必须“看得见”**：脚本/用例名只能写在“执行方式”里，不能当作证据本身。每个 A-xxx 的证据必须包含可见观测产物：关键响应字段值、SQL 结果关键行、以及（涉及数据变更时）变更前/变更后的对比快照；只写“已跑某脚本通过”视为无证据，不通过。
- **证据表达必须面向审阅者**：禁止在对用户的证据中直接使用“脚本变量名/内部计数名”（如 `relation_active_count`、`import_codes_present_count`）。若脚本产出了计数，必须用 **SQL + 结果关键行** 或 **明确的业务实体/记录存在性** 将其改写成用户可理解的证据；接口失败场景必须贴出 `code/errorCode + message` 等关键响应片段，不能只写“返回业务错误”。
- **偏离控制**：实现与修复必须以 PRD/技术设计为准；若确需变更方案或范围，应先回写到相应文档并在日志中记录（但不得“降低验收标准”凑通过）。

## 禁止行为（违反即视为未完成交付）

- **禁止在交付结束时遗留任何由用户完成的验证**：交付结论或对用户的回复中，不得包含任何形式的“需用户执行/用户需做/待用户确认/建议用户…”的验证步骤；交付日志中不得将某 A-xxx 标记为通过却注明“需用户侧执行 XX 后验证”或等价表述。判断标准是**结果**：若用户在看到交付说明后仍需要再执行某操作才能算验收完成，则本次交付无效。
- **禁止在尚有验收项未在本流程内闭环时写交付结论**：只有“执行 + 观测 + 判定 + 证据”均在本流程内完成的验收项才可计为通过；否则视为未通过，不得进入最终交付。
- **客观阻塞时的正确做法**：若因缺鉴权、缺环境、应用未启动等无法在本流程内执行某验收，应**中断**并明确说明需用户提供什么；待用户提供后，在**同一流程内**由你继续执行该验收并记录结果，再考虑交付。不得先写交付结论、再以任何形式把未执行的验收留给用户完成。

## 回复结构（对用户）

每次对话输出应包含：

1. 本次进入的阶段（开发 / 验收 / 修复 / 交付）与覆盖范围（哪些 A-xxx）
2. 已写入/更新的日志位置：`spec/.../05-delivery-log.md`
3. 当前验收结论（通过/不通过）与证据摘要（证据须为本流程内执行所得）
4. 下一步动作（若不通过：立即进入修复并说明计划；若全部在本流程内通过：方可给出最终交付总结；若有客观阻塞：说明需用户提供什么以便本流程继续执行验收）
