# 澄清文档双向同步

本技能直接支撑插件核心理念：**通过用户不断的澄清，从而不断完善所有工程文档**。方向一（提取待澄清）把文档中的不确定点交给用户；方向二（澄清回写）把用户决策写回工程文档，形成闭环，推动文档持续完善。

## 何时使用

- 用户编辑了澄清文档中的某条「澄清结论」，需要把结论写回分析报告/PRD/技术设计/验收清单。
- 用户刚更新了某份文档，需要把其中需澄清的内容抽取到澄清文档。
- 用户问「把待澄清项同步到澄清文档」或「根据澄清结论更新 PRD」。

## 方向一：其它文档 → 澄清文档

1. 针对指定文档（或最近有变更的文档），通读内容。
2. 识别需用户确认或存在二义性的点（范围、优先级、实现选型、验收标准等）。
3. 在 `00-clarifications.md` 中为每个点新增一行（若尚未存在），表格列：ID（C-xxx）、状态（待确认）、优先级、影响范围、归属文档（analysis/prd/tech/acceptance/global）、关联章节、问题/待确认点、用户确认/补充、解决方案。
   - **C-xxx 分配规则**：新增行的 ID 由 agent 分配。在同一需求目录下，取当前澄清文档中已有最大编号（如 C-003）加一得到新号（C-004）；若文档被用户手改过编号导致可能重复，以「不重复 ID」为原则分配下一个可用编号。
   - **去重（防循环）**：若澄清文档中已有**同来源、同问题**的项，只更新描述不重复插入。若某段落/章节已在 00 中有待确认或已确认项且「关联章节」覆盖该处，且本次**未在该段落引入实质性新表述**（如整段重写、新增子节），则**跳过该段落**，不新增待澄清项，避免同一处反复被标为待澄清导致死循环。
4. 保存 `00-clarifications.md` 并告知用户新增了哪些待澄清项。

## 方向二：澄清文档 → 其它文档

1. 读取 `00-clarifications.md`，找出状态为「已确认」且已填写「用户确认/补充」或「解决方案」的项。
2. 根据该项的「归属文档」与「关联章节」定位到目标文档（01/02/03/04）中的对应位置。
3. 结合澄清的「用户确认/补充」与「解决方案」**实质性更新**该文档中所有受影响章节，使正文结论与澄清决策一致；不得仅增加 C-xxx 引用即视为完成。更新后在该文档的 **修订记录** 表中追加一行。
4. **Whole-document review（必须）**：每份文档更新时须**整体回顾整篇文档**，不仅修改与澄清直接对应的段落。通读全篇，检查前后是否一致、与其它章节是否冲突；若有矛盾或更优方案且可自行收敛则直接修正，若需用户决策则追加到 `00-clarifications.md` 为新澄清项、状态为待确认。**若更新 04-acceptance.md**：验收项须保持可测试性（验收步骤可执行、通过标准可断言），澄清导致验收条件变化时须改写为可验证条款。**完成步骤3、4 后写回该文档**（含修订记录）。
5. **回写后必做：对新内容的澄清提取**：澄清内容写回各文档后，**更新后的文档可能包含新的表述，从而产生新的待澄清点**。**仍须**对刚被回写更新的文档（以及若有级联则对级联后更新的每份文档）执行**方向一**（通读、识别），但遵守**兜底防循环**：仅针对**本次更新引入的新表述、新段落、新边界**识别待澄清点；与 00 中已有项**去重**——若**同归属文档且关联章节重叠且问题语义相似**则**不新增行**，只更新已有项描述（若有变化）。经去重后若无确属新点的项，本步**不新增行**即可，避免「回写→提取→同一处又标为待澄清」的循环；**不得跳过本步**（仍须通读并判断）。
6. **回写后的级联**：若回写导致其它文档需要联动修改（如回写更新了 PRD，则技术设计、验收清单可能需同步），须再执行 **cascade-update** 技能：以被回写的文档为「已更新文档」，按前置→后置关系更新其后置文档；cascade 内部会对每份更新后的文档执行澄清提取。回写后无论是否级联，都须完成上一步「对新内容的澄清提取」（并遵守上一步去重与兜底）。
7. **单轮边界（防死循环）**：**单轮**指一次用户发起的回写/同步请求的完整执行（如用户说「把澄清写回 PRD」触发方向二）。单轮内「回写 → 级联（若需）→ 对新内容的澄清提取」执行**一轮即结束本轮回写流程**。新产生的待澄清项交由用户填写后，由用户**显式发起**下一轮回写，不在同一轮内基于刚插入的待澄清项再次自动回写或再次级联，避免无谓循环。

## 路径与模板

- 澄清文档与同需求下其他文档位于同一目录 **spec/<YYYY-MM-DD>/<需求名称>/**，文件名为 `00-clarifications.md`。**目标需求目录**：用户显式指定、或当前编辑文件所在目录、或多需求时按 spec/.active 或向用户确认（见 reference/multi-requirement.md）。结构须与 `templates/00-clarifications.md` 一致（表格：ID、状态、优先级、影响范围、归属文档、关联章节、问题/待确认点、用户确认/补充、解决方案）。

## 澄清文档建议结构

- 待澄清项：来源、问题简述、澄清结论（用户填写）、状态（待确认/已确认，与模板一致）。
- 可选用表格或列表，便于 agent 解析与回写定位。
