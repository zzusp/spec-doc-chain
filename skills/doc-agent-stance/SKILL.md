# 各文档 Agent 表态

本技能实现「第一步：各 agent 先表态」，是向多 agent 协作模式过渡的基础环节。

## 何时使用

- 用户澄清了某条事实（填写了 `00-clarifications.md` 中某项的「用户确认/补充」或「解决方案」），即将执行回写之前。
- orchestrator 在调用 sync-clarification 方向二之前，先调用本技能收集各角色视角的判断。
- 用户显式说「先看看各文档需不需要改」「让各负责人表个态」等。

## 执行步骤

### 步骤 1：读取澄清事实

读取目标需求目录下的 `00-clarifications.md`，定位本轮需要回写的澄清项（状态为「已确认」且已填写结论的项）。

提取关键信息：
- 澄清 ID（C-xxx）
- 澄清内容摘要（问题 + 用户结论）
- 原归属文档与关联章节

**多条澄清的处理规则**：若同时存在多条「已确认」澄清项，**逐条独立执行步骤 2-5**——对每条完整走完「表态 → 决策表 → 冲突检测 → 回写移交」后，再处理下一条。禁止合并多条澄清同时表态，原因：前一条的回写结果可能改变后一条表态时读到的文档内容，合并处理会导致修改方向基于过期文档状态。

---

### 步骤 2：依次激活四个角色表态

按 analysis → prd → tech → acceptance 的顺序，以每个 agent 的**专业身份**（见 `reference/doc-agent-roles.md`）审视这条澄清事实，输出结构化表态。

每个角色的思考流程：

1. **读取自己负责的文档当前内容**（或已知其结构）
2. **对照澄清事实**，判断是否影响文档中的任何章节或结论
   - **重要**：判断基准是「澄清事实对我的文档有无影响」，而不是「澄清的归属文档是不是我」。即使澄清的归属文档标注为其他文档，若其内容与本文档有关联，仍应表态「需要修改」并说明关联理由。
3. **输出表态**，严格遵守格式规范

**表态格式**（每条 ≤5 行）：
```
[{角色}] 需要修改 | 不需要修改 | 待确认
- 理由：引用澄清的哪个关键点，一句话说明
- 影响章节：（仅「需要修改」时填写）章节名或段落标识
- 修改方向：（仅「需要修改」时填写）简述要做什么，不展开
- 对其他文档的影响：（可选）提示可能引发哪些文档联动
```

示例输出（以并发量从100→10000的澄清为例）：
```
[analysis-agent] 需要修改
- 理由：原分析报告对现有系统的「满足」评估需修正
- 影响章节：§5 需求满足性分析、§6 风险与影响
- 修改方向：将单机方案标记为「不满足」，补充分布式缓存分析

[prd-agent] 不需要修改
- 理由：并发量已在 R-08「性能要求」以占位描述体现，数值由技术设计填充

[tech-agent / acceptance-agent 格式相同，填写各自影响章节与修改方向]
```

---

### 步骤 3：汇总形成决策表

orchestrator 汇总四个表态，生成本轮「修改决策表」：

```markdown
## 本轮修改决策（C-xxx：[澄清摘要]）

| 文档 | Agent 结论 | 执行修改 | 影响章节 | 执行时序 | 备注 |
|------|-----------|---------|---------|---------|------|
| 01-analysis.md | 需要修改 | ✅ | §5、§6 | 第1步 | 更新满足性分析与风险评估 |
| 02-prd.md | 不需要修改 | ⏭ | — | — | — |
| 03-tech.md | 需要修改 | ✅ | §4、§6 | 第1步 | 替换缓存方案 |
| 04-acceptance.md | 需要修改 | ✅ | A-007 §4 | 第2步（待03完成） | 更新并发断言数值 |
```

---

### 步骤 4：冲突检测与裁决

检查表态间是否存在**矛盾**，常见情形：

| 矛盾类型 | 处理方式 |
|---------|---------|
| A 说不影响，B 说 A 会联动 | orchestrator 裁决：若 B 的理由成立（如接口变了必然影响 PRD 接口描述），则将 A 列为「需要修改」并注明原因 |
| 某 agent 输出「待确认」 | 暂停该文档的修改，将缺少的信息追加为新澄清项（C-xxx+1），提示用户补充后再执行 |
| 修改方向有依赖顺序 | 按 01→02→03→04 顺序执行，后置文档修改时可参考前置已更新内容 |

**「技术可行性存疑」的处理**：若某 agent 表态「需要修改」，但对修改方向存在技术可行性疑问（例如「当前方案能否满足新指标」），应输出「需要修改」并在修改方向中注明「需验证 XX 前提，建议追加澄清项 C-xxx」，由 orchestrator 将疑问写入新澄清项后继续执行当前修改。**不应输出「待确认」**——「待确认」仅用于表态所需的事实信息不足（如澄清描述不清、缺少关键数据），而非对技术方案的判断存疑。

---

### 步骤 5：输出决策表并判断移交路径

输出本轮决策表，格式：

```
📋 本轮澄清（C-xxx）的修改决策：

✅ 需要修改（3 份）：01-analysis.md、03-tech.md、04-acceptance.md
⏭ 不需要修改（1 份）：02-prd.md

| 文档 | Agent 结论 | 执行修改 | 影响章节 | 执行时序 | 备注 |
|------|-----------|---------|---------|---------|------|
| ... |

[各 agent 表态摘要]
```

**执行时序说明**：若某文档的修改方向依赖另一文档的修改结果（如 04 的断言数值取决于 03 的方案确认），在「执行时序」列中注明「待 03 完成后执行」，回写时按顺序处理。

**全部「不需要修改」的特殊情况**：若四个 agent 均表态不需要修改，则跳过实际写操作，但**必须**在 `00-clarifications.md` 中为该澄清项的「解决方案」列补充「确认：所有文档无需修改，理由：[各 agent 摘要]」，并在澄清文档的修订记录中追加一行，确保可追溯；随后告知用户结论。

**移交路径判断**（以下两条路径互斥，按顺序检查）：

| 条件 | 移交目标 |
|------|---------|
| 决策表存在冲突 / 执行时序依赖 / 新澄清触发（任一） | → 移交 `doc-agent-discussion` 技能进行结构化讨论 |
| 以上均不存在，决策表无争议 | → 直接移交 `sync-clarification` 方向二执行回写 |

---

## 强制约束

- **表态不等于修改**：本技能全程只读文档、输出判断，不执行任何写操作
- **议题锁定**：每次表态只针对当前一条澄清事实，不引入无关议题
- **不替他人决策**：各 agent 只对自己负责的文档表态，提示联动但不替其他文档下结论
- **表态必须有据**：每条表态的「理由」必须引用澄清事实的具体内容，禁止泛泛说「可能有影响」

## 路径约定

- 角色定义见：`reference/doc-agent-roles.md`
- 澄清文档：`spec/<YYYY-MM-DD>/<需求名称>/00-clarifications.md`
- 目标需求目录解析：同 `reference/multi-requirement.md`
