# 澄清结论回写至文档

## 何时使用

- 用户在澄清文档中填写了「已确认」的澄清结论，需要把结论同步回 01/02/03/04 文档
- `doc-chain-orchestrator` 或 `spec-doc-chat` 执行澄清回写操作时调用

## 执行步骤

### 步骤 0：各 agent 表态（回写前必须）

在任何写操作前，先调用 `doc-agent-stance`，以四个文档 agent 角色依次审视本条澄清事实，产出「本轮修改决策表」：

- 若 orchestrator 已在上游完成表态并传入决策表，可跳过步骤 0 直接执行
- 全部「不需要修改」→ 在 00 中为该澄清项补充记录，告知用户，不执行写操作
- 存在「待确认」表态 → 暂停该文档修改，将缺失信息追加为新澄清项，提示用户补充后再发起回写
- 存在冲突 / 执行时序依赖 / 新澄清触发（任一）→ 调用 `doc-agent-discussion` 收敛；收敛后按最终决策表执行
- 其余（无争议）→ 直接按决策表进入步骤 1

**多条澄清的处理**：若同时存在多条「已确认」澄清项，逐条独立执行步骤 0–4，不合并处理。

### 步骤 1：回写文档

1. 读取 `00-clarifications.md`，定位「已确认」且已填写「用户确认/补充」或「解决方案」的项。
2. 根据「归属文档」与「关联章节」定位目标文档中的对应位置。
3. 结合澄清结论实质性更新受影响章节，使文档结论与澄清决策一致；不得仅增加 C-xxx 引用即视为完成。
4. **Whole-document review（必须）**：更新后通读整份文档，确认无内部矛盾；若更新 04-acceptance.md，须保持验收项可测试性（步骤可执行、标准可断言）；并遵守删除约束（只删本用例自建或明确可删数据）。
5. 写回文档，在修订记录表中追加一行。

### 步骤 2：回写后提取新待澄清项（必须执行，不可跳过）

对刚被回写的文档（及步骤 3 级联后更新的每份文档）调用 `extract-clarification`：仅针对本次更新引入的新表述识别新待澄清点，去重后写入 00。经去重后若无确属新点，不新增行即可，但不得跳过此步。

### 步骤 3：级联（若需要）

若回写导致后置文档需联动更新（如回写更新了 PRD，则技术设计、验收清单可能需同步），调用 `cascade-update`（内部会对每份更新后的文档执行 `extract-clarification`）。

### 步骤 4：单轮边界（防死循环）

本轮「回写 → 步骤 2 提取 → 步骤 3 级联」执行一轮即结束。新产生的待澄清项由用户填写后显式发起下一轮回写，不在同一轮内自动再次回写或级联。

## 强制约束

- **回写前必须有决策表**：禁止在无任何角色判断的情况下直接回写所有文档。
- **回写落地**：目标文档的修订记录中必须有本次澄清回写对应条目；正文中归属章节必须已写入澄清结论，禁止仅增加 C-xxx 引用即视为完成。
- **状态仅二值**：00 中「状态」列仅可填写「待确认」或「已确认」。
- **回写后必做提取**：方向二执行完成后必须对刚被回写的文档执行 `extract-clarification`，不得跳过。
