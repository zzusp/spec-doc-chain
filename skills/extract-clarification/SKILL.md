# 从文档提取待澄清项

## 何时使用

- 文档刚被生成或更新，需提取不确定/二义性点写入澄清文档
- `cascade-update` 完成后，对每个更新后的文档提取新待澄清项
- 用户说「从 PRD 提取待澄清」「把不确定的内容加到澄清文档」等

## 执行步骤

1. 通读目标文档，识别需用户确认或存在二义性的内容（范围、优先级、实现选型、验收标准等）。
2. 对每个识别到的点，在 `00-clarifications.md` 中新增一行（若尚未存在该项）：
   - **C-xxx 分配规则**：取当前澄清文档中已有最大编号 +1 得到新号；以「不重复 ID」为原则。
   - **去重（source-sig 优先）**：
     - 构造 `source-sig`：格式建议 `<归属文档>#<H2/H3 标题 slug>#<问题类型>`
     - 若已存在完全相同的 `source-sig`，只在已有行上补充描述，不新增 C-xxx 行
     - 若无相同 `source-sig`，但同归属文档且关联章节重叠，使用语义相似作为二级策略判断是否合并
     - 若该段落已有已确认/待确认项且本次无实质性新表述，跳过该段落
3. 保存 `00-clarifications.md`，告知用户新增了哪些待澄清项（若无新增则说明已检查）。

## 强制约束

- **提取有据**：写入 00 的「问题/待确认点」必须与来源文档中某句表述语义对应，禁止泛泛写无来源的待澄清。
- **必须填写来源信息**：每条新增项的「归属文档」（analysis/prd/tech/acceptance/global）与「关联章节」必须填写。
- **防循环兜底**：若通读后无符合条件的新待澄清点（全部与已有项重复），不新增任何行，仍须回复「已检查，无新待澄清项」，不得跳过此步。

## 路径

- 目标需求目录解析：`reference/multi-requirement.md`
- 澄清文档结构：`templates/00-clarifications.md`
