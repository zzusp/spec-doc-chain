# 记录 Agent 讨论（record-discussion）

## 核心原则

**只记"决策过程"，不记"执行过程"。** 减少冗余的关键是严格区分职责边界：

| 内容类型 | 应记录在 |
|---------|---------|
| 执行步骤与验收证据 | `05-delivery-log.md` |
| 用户澄清问答 | `00-clarifications.md` |
| 多 agent 分歧、假设、被否决方案 | `06-discussion-log.md`（本技能） |
| 最终文档内容 | `01–04-*.md` |

## 触发条件（满足任一则写入）

1. **多方案分歧**：两个或以上 agent 对同一问题提出不同方案，且各有合理理由
2. **重大假设**：agent 在无法向用户澄清时自行做出不可逆选择（如技术选型、接口设计取舍）
3. **方案被否决**：某个被认真讨论过的方案最终未被采纳，值得记录防止将来重复讨论

**不触发的场景（不写入，避免冗余）：**
- 明显的 bug 修复，无需讨论
- 执行步骤的日常沟通（写 05-delivery-log 即可）
- 用户提出的问题（写 00-clarifications 即可）
- 结论已明确写入 01–04 文档且无需额外 why 说明的情况

## 执行步骤

### 写入前检查

1. 判断当前是否满足触发条件；不满足则**不写入**，直接继续执行。
2. 确认目标需求目录（遵循 `reference/multi-requirement.md`）。
3. 检查 `06-discussion-log.md` 是否已存在：
   - 存在 → 追加新讨论记录（不重写历史）
   - 不存在 → 用 `templates/06-discussion-log.md` 初始化，然后追加

### 写入规则

- **D-ID**：同需求目录内已有最大编号 +1；若文件刚创建则从 D-001 开始。
- **参与方**：写 agent 名称（如 `delivery-orchestrator` / `verifier`）；若是单 agent 自决，写「单方决策」并注明 agent 名。
- **结论必须含 why**：不能只写选了哪个方案，必须说明理由（哪怕一句话）。
- **被否决方案**：若有，必须在「被否决方案备注」中写明否决理由，防止将来重走弯路。
- **对文档的影响**：若讨论结论影响 01–04 中的某份文档，必须同步更新该文档或追加到 `00-clarifications.md` 待用户确认。
- **快速索引必须同步更新**：每次新增详情，同步在文件顶部「快速索引」表加一行。

### 写入后

- 若讨论结论影响验收项（A-xxx），通知 `delivery-orchestrator` 同步更新 `04-acceptance.md`。
- 若有需用户决策的点，追加到 `00-clarifications.md` 并提示用户。
- **不触发级联**：讨论记录本身不驱动文档级联，只是辅助信息；文档变更由对应技能负责。

## 反冗余检查（写入前自查）

- [ ] 这条信息是否已经完整体现在 01–04 或 05 文档中？若是 → 不写
- [ ] 这是执行日志还是决策记录？若是执行日志 → 写 05，不写 06
- [ ] 这是用户澄清还是 agent 内部决策？若是用户澄清 → 写 00，不写 06
- [ ] 讨论结论的「why」是否写清楚了？若无 why → 补充后再写入
