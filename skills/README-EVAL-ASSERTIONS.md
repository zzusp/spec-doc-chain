# 评测断言加强指南

当前 evals 的断言偏「存在性」和「结构有即可」，baseline（不用技能、仅靠 reference/templates）也能满足，导致 with_skill 与 without_skill 通过率都是 100%，**无法区分技能是否真正被遵循**。加强断言的目的，是让断言**可验证、且只有按技能要求做才会通过**。

---

## 零、评测结论后的两类优化

跑完评测、看完 Review 后，「优化」可以指两件事，不要混在一起：

| 类型 | 做的是什么 | 改的是啥 |
|------|------------|----------|
| **优化评测** | 让「判题标准」更严、更能区分 with_skill / without_skill | 加强断言、增加 evals、把 Review 里发现的问题记成新断言或新 eval；**不改** SKILL.md |
| **优化技能** | 让「技能本身」在下次跑时表现更好 | 根据 Review 里看到的漏步骤、格式不符、没按 spec，去**改各技能的 SKILL.md**（步骤、约束、示例等），再跑 iteration-2 看通过率 |

两者可以同时做：先加强断言 / 加 evals 把「哪里不行」量出来，再根据同一份 Review 去改 SKILL.md，下一轮用更严的断言再看一次。

---

## 一、设计原则

1. **可客观验证**：能通过读文件内容/结构做出 pass/fail 判断，不依赖主观感受。
2. **与技能强绑定**：技能里明确要求的步骤、格式、约束，写成「只有做到 X 才通过」的断言。
3. **避免泛泛而谈**：不写「内容充实」「表述清晰」这类难以界定的要求；改为「某节非空」「某列取值来自枚举」「某处出现某关键词」等。
4. **适度粒度**：一条断言只查一件事，方便定位失败原因；重要约束可拆成多条。

## 二、各技能加强断言建议

### 1. generate-doc-chain

**技能要点**：顺序 01→02→03→04→00、先读全局记忆、02 引用 01、03 与 PRD 一致、04 可测试、每份 whole-document review、待澄清写入 00。

| 断言（建议新增/替换） | 目的 |
|------------------------|------|
| 01 的「结论」节、02 的「需求范围」或等价节、03 的「实现目标」或等价节**非空且非仅占位符**（如非仅「待补充」） | 确保有实质内容，而非只套模板 |
| 02 正文中**明确引用 01**（如「见分析报告结论」「01-analysis 中…」或引用 01 的章节名） | 强制前置引用，体现顺序与依赖 |
| 04 中至少一条验收项同时包含**「验收步骤」与「通过标准」**（或表格中对应列），且通过标准**不包含**「功能正常」「体验良好」等不可断言表述 | 落实技能要求的可测试性 |
| 若项目存在 `spec/00-global-memory.md`，01 或 02 中**提及全局记忆或项目级约束**；若不存在，则生成过程中**创建了** `spec/00-global-memory.md` 或回复中说明已创建 | 强制遵守「先读/先建全局记忆」 |
| 00 中「归属文档」列取值**仅为** analysis / prd / tech / acceptance / global 之一（与模板枚举一致） | 避免乱填，与模板契约一致 |
| 回复中**列出生成的文件清单或下一步建议**（如待澄清条数、建议回写） | 体现技能要求的「列出清单 + 推动澄清」 |

---

### 2. analyze-existing-project

**技能要点**：基于真实代码扫描、01/02/03 与代码对应、无法推断的写澄清或「待业务确认」。

| 断言（建议新增/替换） | 目的 |
|------------------------|------|
| 01 中**至少 2 处**出现具体路径或文件名（如 `reference/*.md`、`templates/01-*.md`、`skills/`、`SKILL.md` 等） | 证明基于实际项目结构，非泛泛描述 |
| 02 中出现**「由现有系统逆向归纳」或「基于代码/现状反推」**等等价表述 | 与技能要求的标注一致 |
| 03 中技术栈或模块描述与项目**一致**（如本仓库有 Markdown、无后端服务则不应写「API 服务」为主体） | 可验证的「基于扫描结果」 |
| 01/02/03 任一份中**不含**明显与项目无关的虚构模块名（如凭空出现「订单服务」「支付网关」且项目无对应目录/文件） | 减少臆造，提高区分度 |

---

### 3. cascade-update

**技能要点**：归纳变更点、按前置→后置更新、每份更新后澄清提取、去重。

| 断言（建议新增/替换） | 目的 |
|------------------------|------|
| 03 中出现的**功能或需求表述与 02 中用户给出的变更一致**（如 prompt 中写「用户登录、导出 PDF」，03 中应出现这两项或明确对应） | 级联一致性，非随意写 |
| 00 中新增澄清项的「归属文档」「关联章节」**非空**，且能与 03 或 04 的某小节/表格对应 | 澄清提取有据可查 |
| 04 中验收项（A-xxx）与 03 中功能/设计点**可对应**（如 A-001 对应用户登录、A-002 对应导出 PDF） | 前后追溯，体现级联 |
| 03、04、00 的**修订记录**中各有至少一行与本次级联/澄清相关的描述 | 落实「每更新则追加修订记录」 |

---

### 4. sync-clarification

**技能要点**：方向一提取待澄清、方向二回写且 whole-document review、回写后必做澄清提取、状态仅待确认/已确认。

| 断言（建议新增/替换） | 目的 |
|------------------------|------|
| 方向一：00 中 C-xxx 的「问题/待确认点」与 02（或来源文档）中**某句表述语义对应**（如 02 写「是否支持多语言待定」，00 中应有对应问题） | 提取准确，非泛泛一条 |
| 方向二：回写后 02 的**修订记录**中有一条与「澄清回写」或 C-xxx 相关；02 正文中**对应章节已写入澄清结论**（如「高优先级」） | 回写落地到正文+修订记录 |
| 00 中「状态」列**仅出现**「待确认」或「已确认」（无「已闭合」「已处理」等） | 与模板与技能用词统一 |
| 回写后若 00 有新增行，则新增行的「归属文档」与刚被回写的文档一致；若无新增，则回复或 00 中能看出「已做澄清提取」 | 落实「回写后必做澄清提取」 |

---

### 5. spec-doc-chat

**技能要点**：统一入口、解析意图并调度子技能、回复含下一步建议。

| 断言（建议新增/替换） | 目的 |
|------------------------|------|
| 生成链场景：回复中**明确出现**「按序生成」「generate-doc-chain」或 01→02→03→04→00 的生成顺序说明 | 体现「解析为按序生成并调度」 |
| 生成链场景：回复中**列出生成的文件路径或清单**，且**含下一步建议**（如待澄清、回写、级联） | 与技能要求的回复结构一致 |
| list 场景：summary 中需求目录路径格式为 **spec/YYYY-MM-DD/名称**，且与 `spec/` 下实际子目录**一致**（条数、路径可核对） | 证明按 multi-requirement 规则扫描 |
| list 场景：**提及** spec/.active 或「当前激活需求」（若不存在则说明「未设置」） | 与技能约定一致 |

---

### 6. spec-delivery-chat

**技能要点**：验收在本流程内执行并判定、证据落盘、责任不转嫁用户。

| 断言（建议新增/替换） | 目的 |
|------------------------|------|
| 05-delivery-log 中**验收项状态总表**包含本需求所有 A-xxx，且对应状态为「通过」或「不通过」（非空白） | 落实「状态总表必须更新」 |
| 05 或 delivery-rounds 的 round 文件中**包含本轮执行的命令或可观测输出摘要**（如 `Test-Path README.md` 或命令结果） | 证据落盘，可复现 |
| 05 全文**不出现**「需用户执行」「请用户验证」「待用户确认后再验收」等将验收责任转嫁的表述 | 落实「验收责任不可转嫁」 |
| 若有 round 文件，则主日志的**轮次索引**中有一条指向该 round 文件（路径或名称一致） | 索引与详情一致 |

## 三、落地方式

- **eval_metadata.json**：在对应 `eval_metadata.json` 的 `assertions` 数组中，用上表中的断言**替换或追加**现有条目；保持每条一句话、可独立判 pass/fail。
- **evals.json**：在 `evals.json` 的 `expected_output` 或对应 eval 的 `expectations`（若使用）中，可与上述断言对齐描述，便于人工对照。
- **评分**：grader 或脚本按「证据在 outputs 中是否存在」判定；若某条难以自动判断，可保留为「人工检查项」并在 grading 时注明。

按上述方式加强后，再跑一轮 with_skill / without_skill，预期会出现「with_skill 通过更多、without_skill 部分失败」的差异，从而体现技能价值。
