# 流程环节与流转总览

本文档汇总插件内各流程的入口、步骤、出口及与其它环节的衔接，便于核对流转是否完善。

## 一、目标需求解析（所有写操作前置）

任一涉及「写文档或读需求目录」的操作，须先确定**目标需求目录**，顺序见 [multi-requirement.md](multi-requirement.md)：

1. 用户显式指定需求名称或路径  
2. 当前编辑文件所在需求目录  
3. spec/.active（存在且指向有效目录）  
4. spec 下仅有一个有效需求目录  
5. 否则：向用户确认并列出已有需求（可触发 doc-chain-list）

**无任何需求目录时**：list 返回空并提示可新建；生成链由用户提供需求名称后创建 `spec/<日期>/<名称>/`。

---

## 二、按序生成文档链（generate-doc-chain）

| 环节 | 说明 |
|------|------|
| **入口** | 用户说「生成整套文档」「从分析报告/PRD/技术设计开始生成」等；命令 `doc-chain-generate-all` |
| **前置** | 确认需求名称与日期；若从非 01 开始，须确认前置文档已存在 |
| **步骤** | 读全局记忆 → 从起始文档起按 01→02→03→04→00 顺序生成；每份生成前读已存在前置与已确认澄清；**写回前须 whole-document review**；生成后有待澄清则写入 00（00 若不存在则在生成到澄清文档时创建并写入积累项） |
| **出口** | 列出生成的文件、待澄清项数量；建议下一步（填写澄清后回写） |
| **衔接** | 生成完成后可写入 spec/.active；后续可进入「澄清回写」或「级联更新」 |

---

## 三、级联更新（cascade-update）

| 环节 | 说明 |
|------|------|
| **入口** | 用户说「我改了 PRD，请同步后面文档」等；命令 `doc-chain-cascade-after-edit` |
| **前置** | 确认被更新的文档及路径；目标需求目录（常由当前编辑文件反推或 .active） |
| **步骤** | 归纳变更点 → 按 01→02→03→04 的前置→后置关系依次更新后置文档（**每份更新时须 whole-document review**：整体回顾该份文档，非仅改直接相关段落；有矛盾可自行收敛则修正，需用户决策则写入 00）→ 每更新一份则修订记录追加一行并执行澄清提取（写入 00） |
| **出口** | 列出已级联更新的文件、新增待澄清项；建议下一步（填写澄清后回写） |
| **衔接** | 仅改 04 时无后置文档，但仍执行澄清提取；若某后置文档不存在可提示先生成（generate-doc-chain） |

**说明**：cascade-update 内部已含「对每个更新后文档做澄清提取」，命令层无需再单独调用 sync-clarification 方向一。

---

## 四、澄清双向同步（sync-clarification）

### 方向一：其它文档 → 澄清文档（提取待澄清）

| 环节 | 说明 |
|------|------|
| **入口** | 用户说「把 PRD 里需要澄清的点抽到澄清文档」；或由 generate/cascade 内部执行 |
| **步骤** | 通读指定/最近变更文档 → 识别待确认或二义性点 → 在 00 中新增行（C-xxx 分配规则），同来源同问题不重复插入 |
| **出口** | 告知新增了哪些待澄清项；建议用户填写后执行回写 |

### 方向二：澄清文档 → 其它文档（澄清回写）

| 环节 | 说明 |
|------|------|
| **入口** | 用户说「把澄清结论写回 PRD」；命令 `doc-chain-sync-clarification`（回写分支） |
| **步骤** | 读 00，找状态为已确认且已填结论的项 → 按归属文档与关联章节定位 → 实质性更新目标文档并追加修订记录 → **必做 whole-document review**：整体回顾整篇文档（非仅改直接对应段落），有矛盾可自行收敛则修正，需用户决策则追加到 00 → **必做**：对刚被回写更新的文档（及若有级联则对级联后更新的每份文档）执行**澄清提取（方向一）** → 若影响其它文档则再执行 cascade-update |
| **出口** | 列出回写与级联更新的文件、澄清中已确认的项、**本次新产生的待澄清项**；建议下一步 |
| **衔接** | **牢记**：澄清内容写回各文档后，更新后的文档可能含有新的待澄清内容，必须再提取并写入 00，形成「回写 → 新待澄清 → 用户再澄清」循环；回写后若需级联则再调用 cascade-update（其内部也含澄清提取） |

---

## 五、分析已有项目（analyze-existing-project）

| 环节 | 说明 |
|------|------|
| **入口** | 用户说「分析当前项目并生成分析报告、PRD、技术设计」；命令 `doc-chain-analyze-project` |
| **步骤** | 确认代码根目录与需求名称 → 扫描与归纳 → 按模板生成 01、02、03 → 若有二义性则创建/更新 00 并写入待澄清项 → 可选：继续生成 04 并完善 00（00 已存在则不覆盖，仅补充） |
| **出口** | 列出生成文件、分析结论摘要；建议下一步（补充澄清后回写或生成完整链） |
| **衔接** | 与 generate-doc-chain 配合时，相当于 01～03 已生成，可选补 04 与澄清；生成后可写 spec/.active |

---

## 六、多需求管理（list / set-active）

| 环节 | 说明 |
|------|------|
| **list** | 按需求目录识别规则扫描 spec/，列出有效需求目录及当前 .active；无任何需求目录时返回空并提示可新建 |
| **set-active** | 解析目标需求 → 确认目录存在 → 写入 spec/.active（推荐带 spec/ 前缀） |
| **衔接** | 生成或分析完成后可顺手 set-active；多需求时写操作前可先 list 或 set-active 明确目标 |

---

## 七、统一入口与编排（spec-doc-chat / doc-chain-orchestrator）

- 将自然语言解析为上述任一流程或组合。
- 多步可串联：如先 cascade-update 再提示用户澄清并执行 sync-clarification 方向二。
- 回复须包含「下一步建议」，围绕澄清循环（待澄清 → 用户填写 → 回写 → 必要时级联）。

---

## 八、交付闭环入口（spec-delivery-chat / delivery-orchestrator）

当用户已完成文档链并进入实现阶段时，使用交付入口执行完整闭环：

| 环节 | 说明 |
|------|------|
| **入口** | 用户说「按验收清单开始开发/交付」「跑验收不通过就修到通过」等；命令 `spec-delivery-run`；技能 `spec-delivery-chat` |
| **前置** | 解析目标需求目录；读取 `02-prd.md`、`03-tech.md`、`04-acceptance.md`（必要时读取 `00-clarifications.md` 的已确认项） |
| **步骤** | 开发实现（建议按 A-xxx 做 TDD）→ 真实执行测试/验收 → 结论（通过/不通过）→ 不通过则定位根因并修复 → 回到验收重复直至全部通过；全过程持续追加 `05-delivery-log.md`。**若用户反馈与预期有偏差/需调整**：先记录为变更请求并对齐更新 `02/03/04`，再进入新一轮闭环直到用户满意且验收通过 |
| **出口** | 全部 A-xxx 通过且用户确认满意后，给出最终交付总结与逐项证据，并在 `05-delivery-log.md` 写明结束时间与通过依据 |

---

## 九、流程兜底（防死循环）

- **澄清提取去重**：提取待澄清项时，若 00 中已有**同来源、同问题**的项，只更新描述不新增行。若某段落已有对应澄清项且该段落**无实质性变更**，跳过该段落。若与已有项**同归属文档且关联章节重叠且问题语义相似**，则不新增行，避免同一处反复被标为待澄清；同一章节内**不同问题**（如范围 vs 优先级）仍可新增。
- **回写后提取**：回写后**仍须执行**「对新内容的澄清提取」（通读、识别），仅插入**本次更新引入的新表述**且经去重后确属新点的项；若无则本步不新增行。不得跳过本步。
- **单轮边界**：**单轮** = 一次用户发起的回写/同步请求的完整执行。单轮内「回写 → 级联（若需）→ 澄清提取」执行**一轮即止**；新待澄清项由用户填写后，由用户**显式发起**下一轮回写。
- **级联提取**：cascade 时对每份更新文档做澄清提取须与 00 已有项去重（同归属+关联章节重叠且问题语义相似则不新增），同一段落不重复插入。

---

## 九、流转自检清单

- [ ] **生成链**：从 01 开始或从中间开始时，前置文档存在性已约定；00 不存在时在生成到澄清文档时创建并写入待澄清项。
- [ ] **Whole-document review**：每份文档更新时须整体回顾整篇文档，不仅改直接对应段落；有矛盾可自行收敛则修正，需用户决策则追加到澄清文档。
- [ ] **级联**：每个后置文档更新时须 whole-document review，更新后均执行澄清提取；仅改 04 时也执行澄清提取；后置文档不存在时有提示。
- [ ] **澄清回写**：回写时须 whole-document review；回写后**必对已更新文档做澄清提取**；若影响其它文档须再执行 cascade-update，且 cascade 内部已含澄清提取。
- [ ] **分析项目**：00 可由步骤6 先创建；步骤7 补 04 与澄清时不覆盖已有 00 内容。
- [ ] **目标需求**：无需求目录时 list 空并提示新建；set-active 要求目标目录存在。
- [ ] **命令与技能**：cascade-after-edit 不重复调用澄清提取；sync-clarification 命令在回写后明确调用 cascade-update。
- [ ] **防死循环**：澄清提取与已有项去重、同段落不重复插入；回写后提取仅针对本次新表述且去重；单轮内回写+级联+澄清提取一轮即止，下一轮由用户显式发起。
