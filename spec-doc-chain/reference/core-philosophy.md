# 核心理念

本插件的核心理念是：

**通过用户不断的澄清，从而不断完善所有工程文档。**

## 含义

- 工程文档（分析报告、PRD、系统技术设计、验收清单）在生成或级联更新过程中，会自然产生**待确认、待决策、存在二义性**的点。
- 这些点应被提取并写入**澄清文档**（`00-clarifications.md`），由用户补充「用户确认/补充」与「解决方案」。
- 用户填写澄清结论后，结论应**回写**到对应的分析报告、PRD、技术设计或验收清单，使正文与用户意图一致。
- **重要**：回写会改写文档正文，**更新后的文档可能包含新的表述，从而产生新的待澄清点**（如新边界、新选型、新验收条件等）。因此回写后必须对已更新的文档再次执行「提取待澄清项」并写入澄清文档，再由用户补充，形成**循环**。回写或级联后均须做此步。
- 该循环持续进行，直至文档与用户意图一致、待澄清项闭合，文档质量得以持续提升。

## 澄清文档的角色

澄清文档是**用户与整套工程文档之间的枢纽**：

- **方向一**：从各文档中提取待澄清项 → 写入澄清文档（用户在此填写）。
- **方向二**：从澄清文档中读取已确认的结论 → 回写并更新相关文档。

因此，**提取待澄清**与**澄清回写**不可省略；每次生成或更新文档后应主动检查并写入待澄清项，用户填写后应主动建议执行回写，推动循环。**防死循环兜底**（去重、单轮边界）仅用于避免同一处反复标为待澄清或同一轮内无限链式执行，**不改变**「回写后必做澄清提取」等要求。

## 在行为上的落实

- **生成文档后**：若有需用户确认或二义性内容，必须在澄清文档中新增对应行，并在回复中提示待澄清项及建议下一步。
- **级联更新后**：同样提取新待澄清项写入澄清文档，回复时建议用户填写后执行回写。
- **用户填写澄清后**：执行回写并更新相关文档；**回写后须对更新后的文档做澄清提取**（更新后的内容可能产生新待澄清点，写入澄清文档），若需级联则再执行级联并对级联后文档做澄清提取；回复时提示新待澄清项及下一步。
- **统一入口与编排**：回复中应始终围绕「待澄清项 → 用户澄清 → 回写」给出下一步建议，不跳过澄清循环。
