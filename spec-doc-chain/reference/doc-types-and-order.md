# 文档类型与前置顺序

本插件管理的文档类型及推荐生成/更新顺序如下。顺序即前置关系：前者更新后，需考虑对后者的级联更新。

| 顺序 | 文档类型       | 文件名（同目录下固定） | 说明 |
|------|----------------|------------------------|------|
| 1    | 分析报告       | `01-analysis.md`       | 需求/现状分析，结论与建议 |
| 2    | PRD            | `02-prd.md`            | 产品需求文档，依赖分析报告 |
| 3    | 系统技术设计   | `03-tech.md`           | 技术方案/架构与实现，依赖 PRD |
| 4    | 验收清单       | `04-acceptance.md`     | 可验收条目（A-xxx），依赖 PRD 与技术设计 |
| 5    | 澄清           | `00-clarifications.md` | 待澄清项与澄清结论，与所有文档双向同步 |

**路径约定**：所有文档按 **日期 + 需求名称** 存放，**推荐**格式为 `spec/<YYYY-MM-DD>/<需求名称>/`（与 spec-agent 一致）。详见 [path-convention.md](path-convention.md)。

**文档模板**：各文档类型均有统一模板框架（与 spec-agent 章节对齐），生成时须按模板填充。模板位于插件 `templates/` 目录：
- 分析报告 → `templates/01-analysis.md`
- PRD → `templates/02-prd.md`
- 系统技术设计 → `templates/03-tech.md`
- 验收清单 → `templates/04-acceptance.md`
- 澄清 → `templates/00-clarifications.md`

**澄清文档与核心理念**：本插件的核心理念是**通过用户不断的澄清，从而不断完善所有工程文档**。澄清文档是枢纽：任何文档更新后，需用户确认或二义性的内容应自动插入澄清文档；用户在澄清文档中填写/修改澄清结论后，应回写并更新相关文档。**每份文档更新时须 whole-document review**（整体回顾整篇文档）；回写后须对已更新文档再做澄清提取（更新后的内容可能产生新待澄清项）。该「提取待澄清 → 用户澄清 → 回写文档」的循环持续进行，直至与用户意图一致。

**需求追溯（可选）**：与 spec-agent 配合时，可在 PRD/技术设计/验收清单中使用 R-xx 需求编号，使验收项（A-xxx）能追溯到 PRD 与技术方案中的需求与设计，便于实现阶段 TDD 与追溯。

**统一入口**：用户通过单一对话入口即可完成「按序生成、更新某文档、级联更新、澄清同步、对已有项目进行分析并生成文档、查看状态」等全部操作。
